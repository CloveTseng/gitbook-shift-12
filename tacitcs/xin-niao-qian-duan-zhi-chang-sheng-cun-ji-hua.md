# 新鳥前端職場生存計畫

`w1`

### 企業文化與「體質」的初步感受

這段時間不需要急著上手複雜代碼，而是觀察環境是否健康。

* 入職引導（Onboarding）： 公司是否有清楚的交接文件、環境設定教學？還是放任你自生自滅？&#x20;
  * `沒有交接文件，但主管是走扁平化管理，大家有意見都隨時可以提出討論`
* 團隊氛圍： 同事之間討論問題是互相幫忙，還是推諉責任？
  * &#x20;`目前觀察中`
* 開發流程： 是否有基本的 Git 版本控制、專案管理工具（如 Jira/Trello）？如果連基本的 Commit 規範都沒有，未來維護會很痛苦。
  * &#x20;`公司使用 gitlab 跟自己寫的專案管理工具`

### 第一個月：工作節奏與「技術棧」的契合度

這時你開始接觸真實專案，是判斷技術成長空間的最佳時機。

* 代碼質量： 觀察現有的 TypeScript 寫法（是寫滿 `any`，還是有嚴謹的 `type` 定義？）以及 Nuxt.js/Vue 的架構是否符合 Composition API 的規範。
* 溝通效率： 與主管、PM 溝通需求時是否順暢？需求是否朝令夕改？
* 壓力測試： 觀察團隊加班的頻率與原因。是因為產品上線，還是因為流程混亂導致的無意義內耗？

### 三個月（試用期結束）：核心價值與「未來性」

這是決定長期留任的最關鍵點。

* 成長價值： 在這三個月內，你學到的東西是否符合你的職涯目標（例如：是否有機會實踐你正在學的 SSR 或自動化流程）？
* 反饋機制： 主管是否會針對你的表現給予具體的建議，而非只有模稜兩可的評價？
* 直覺（Gut Feeling）： 每天早上起床想到要上班，你是感到期待、平靜，還是極度的焦慮與抗拒？

* 了解公司文化與組織架構 `w1`
  * 搞清楚「誰是誰」： 不只是技術主管，還要知道 PM（產品經理）、QA（測試）以及核心開發者是誰。觀察遇到問題時，大家傾向找誰討論。
  * 同步溝通頻率： 了解團隊使用 Slack、Discord 或 Teams 的習慣。觀察大家在什麼時候會發起討論，非同步溝通的節奏如何。
  * 閱讀內部文件： 找尋公司的 Wiki、Notion 或 Confluence。了解公司的產品願景與核心價值，這能幫助你在寫代碼時更理解業務邏輯（Business Logic）。
* 理解公司技術架構與開發流程 `w2~w4`
  * 熟悉公司的環境建置習慣與 git flow
  * CI/CD 流程
  * TypeScript：專案怎麼定義 `type`
  * Vue：觀察專案的目錄架構（`components/`, `composables/`, `middleware/`），注意團隊如何拆解 SFC 以及 Composition API 的封裝邏輯
  * API 串接： 檢查專案中 `axios` 或 `ohmyfetch` 的封裝方式，了解攔截器（Interceptors）如何處理 Token 或報錯。
* 獨立開發能力 `w9~w12`
  * 爭取拿到獨立開發專案
  * 善用 Obsidian： 將公司專案中常見的 `type` 定義方式、特殊的開發規範記錄在 Obsidian 中，建立自己的「公司技術知識庫」。
